## 2026-07-17 — The demo cursor teleported because only transform had a transition #incident #decision

## 2026-07-16 — The trailer device is a laptop, and it had to be modeled from scratch #decision

## 2026-07-16 — Puppeteer clicked the Google button because both forms say submit #incident

The screenshot run for the trailer logged in as the demo volunteer, or so it claimed: `page.click('button[type="submit"]')` on the login page actually hit the Google OAuth button, because the OAuth form sits above the email form and both end in a submit button. Half the "authenticated" captures came back as login redirects. The demo suite never hit this because Playwright's `getByRole('button', { name: /Sign in/i })` selects by accessible name, not by position. Scoping the selector to `form.auth-form button[type="submit"]` fixed the volunteer shots; the NGO login then stalled anyway until it got a fresh incognito context, since the Appwrite client session from the previous login survives cookie deletion. Lesson: on a page with two forms, `button[type=submit]` is a coin flip, and signing out is not the same as starting clean.

## 2026-07-16 — Watched the demo back and found the payoff shot below the fold #incident

The closed-loop recording passed every assertion, so it was ready to ship. Extracted frames to actually look at it and the final beat, the entire reason the scenario exists, had the earned badge tile sitting just under the viewport: the "Badge vault" heading and "1 earned" were in shot, the tile itself wasn't. `scrollIntoViewIfNeeded()` moves the minimum distance, so anchoring on the section heading parks that heading at the bottom edge and leaves everything below it off-screen. `toBeVisible()` stayed green throughout because the element was rendered and unobscured, which is all it actually checks. Swapped the anchor to the tile and the assertion to `toBeInViewport()`, which is the one that would have caught it, and did the same to the locked-vault shot at the top so the before/after framing matches. Lesson: for anything whose output is a video, the exit code isn't the test, the frames are.

## 2026-07-16 — Seeding grew two sign-in-able accounts, because the loop can't be filmed without them #decision

Recording claim → submit → approve → badge needs two accounts that can sign in through the real UI, and `scripts/seed.ts` only ever made one NGO user with a throwaway random password, under a comment reading "no one signs in as this account; we mutate via admin SDK only." Extended it to set a real password on that user, add a demo volunteer, and create a `task-completion` badge definition so approval has something to mint. The NGO has to be that seeded user specifically: `/api/claims/[id]/approve` 403s any reviewer who doesn't own the task, so a separate reviewer account can't stand in. The password comes from `SEED_DEMO_PASSWORD` and the whole block no-ops when it's unset, because hardcoding it would commit an NGO-role login that can approve claims and mint badges, pointed at the same Appwrite project the live site runs on. The other half was making the seed reset the volunteer's claims and badges every run: `processBadgeAwards` skips any label the user already holds, so a second recording approves the claim and reveals no badge while every assertion still passes. Named the badge "First Mission" to match one of the locked placeholders already in the vault, so the approval visibly unlocks that exact tile. Wired `bun run seed` into `bun run demo` so nobody has to remember. Lesson: fixture state with a shelf life should be reset by the script that depends on it, not by a line in a README that someone reads once.

## 2026-07-16 — The demo login put the password in the URL when hydration lost the race #incident

Building the closed-loop spec, ran the login helper with slowMo at zero and watched Playwright navigate to `/login?email=demo%40micromatch.app&password=Demo-...`. The form had done a plain native GET: `on:submit={handleEmailSignIn}` is the thing that calls `preventDefault()`, and the click landed before Svelte hydrated, so nothing intercepted it. At the normal `DEMO_SLOWMO=1200` this never reproduces, since every action pauses over a second and hydration always wins, which is precisely why it could have sat there indefinitely. It only surfaced because the probe ran at zero pacing. Fixed by waiting on `<dotlottie-player>`, which AuthBrandPanel only renders after onMount's dynamic import resolves, so its presence proves hydration finished. The same bug was already sitting in 02-signup-flow, where clicking the role picker pre-hydration silently does nothing and the form never appears; it had gone unnoticed because slowMo hid it there too. Lesson: the pacing that makes a demo readable also papers over real races, so the honest place to run a demo spec is at zero slowMo, not at recording speed.

## 2026-07-16 — Demo tasks weren't deleted, they'd just aged out of the feed #incident

Pre-flighting the marketing/demo trailer shoot, checked the production site and found `/tasks` showing the "You're too fast!" empty state, the same one local dev shows with no seed data. Ran `bun run seed` expecting it to backfill, and it reported "8 already present": the demo tasks were still sitting in Appwrite the whole time. Something was hiding them from the feed, not deleting them. Traced it to `filterTasksForFeed()` in `appwrite.ts`, which auto-archives any task whose `lastActivityAt` is more than 30 days stale (a feature meant to hide dead listings). The 8 demo tasks were seeded 2026-05-09 and, being demo tasks, never got organic claims to keep `lastActivityAt` fresh, so they silently aged past the 30-day line with zero errors anywhere: not in Vercel runtime logs, not in the seed script's own output, since its idempotent "skip if title exists" path never touched the row again after creation. Fixed `ensureTasks()` to look existing rows up by id instead of just title, and to call `updateRow` on their `status`/`lastActivityAt` every run, so re-seeding revives stale demo data instead of being a no-op. Re-ran it and the 8 tasks are back live. Lesson: an idempotent "skip if exists" script and a silent auto-archive filter are each fine on their own, but stacked together they produce a bug with no error message and no code path that ever surfaces it. Seed data has a shelf life now, and this exact wall will get hit again before every future demo unless someone reseeds first.

## 2026-07-16 — CI had been red for two days on a compiler bump nobody was watching #incident

Checked whether CI needed updating after the auth fixes and found it had been failing since commit e4c2ed7 two days earlier: "bump dependencies to latest" had pulled `typescript` up to 7.0.2, the Go-ported rewrite, and both `Type-check` and `Lint` jobs were crashing before they even reached a file. svelte-check reaching into `typescript.sys` got `undefined`, and typescript-eslint's `typescript-estree` failed the same way. Confirmed via `gh run list` that the two most recent pushes to main both show `failure`, and the logs matched a local repro exactly. The real tell was in typescript-eslint's own package metadata: its published peerDependency range is `typescript: ">=4.8.4 <6.1.0"`, so TS7 was never going to work no matter which version got installed. Pinned back to `^5.9.3`, the newest release inside that window, and both jobs went green: svelte-check at 0 errors, eslint at 0 errors and 140 pre-existing warnings that don't fail the build. One unrelated find along the way: eslint never had `.vercel/**` or `coverage/**` in its ignore list, invisible in CI since both are gitignored, but a wall of noise for anyone linting locally after a build. Added them. Lesson: "bump dependencies to latest" is not a decision, it's a coin flip against whatever the rest of the ecosystem has caught up to.

## 2026-07-16 — hooks.server.ts had a fallback that let anyone become an NGO for free #incident

A security review found that `hooks.server.ts` fell back to the client-writable `mm_role` cookie to populate `event.locals.userRole` whenever the HttpOnly `mm_session` cookie was missing or expired, and `getUserRole()` treats `locals.userRole` as authoritative. That meant `Cookie: mm_role=ngo`, sent with zero authentication, was enough to pass every `role === 'ngo'` gate: approve or reject any claim, run `/api/tasks/maintenance` (which sweeps every org's tasks, not just the caller's), claim tasks anonymously. Most NGO-only routes survived anyway because they also require `session.user.id` for ownership lookups, and the forged cookie never populates that. Claims approve/reject didn't do an ownership check at all, though, so two real logged-in NGOs could already approve or reject each other's claims and mint badges on tasks they don't own; the cookie bug just made the same holes reachable by logged-out attackers too. Fixed by deriving `locals.userRole` only from a validated `mm_session` lookup or `getUserRole()`'s JWT/token fallback, adding task-ownership checks to both claim review endpoints, and moving `/api/tasks/maintenance` from "any NGO" to `isUserAdmin`. `mm_role` still exists and is still read, just client-side now, in Sidebar.svelte and +layout.svelte, for optimistic rendering before the session round-trip lands. Lesson: a code comment promising a cookie is "not used for authorization" is a bug report waiting to happen if nothing actually enforces it.

## 2026-05-12 — Wrote a positioning doc instead of pretending we're Idealist #decision

Added `docs/positioning.md` to pin down what MicroMatch actually is, because the README kept drifting toward generic "volunteer platform" language. The honest framing: Idealist is a directory you leave to email an org; MicroMatch is a marketplace where the work happens on-platform — claim → submit → approve → badge is one closed loop. ## 2026-05-09 — The Safari ITP OAuth loop that took five commits to kill #incident

Google sign-in was sending Safari users into an infinite loop: OAuth → /profile → save → /login → OAuth, forever. The root cause was Intelligent Tracking Prevention stripping the session cookie set by `cloud.appwrite.io` as third-party, so every `account.*` call returned 401. We thrashed through it — first bounced to /login on a load-time 401 (broke Google OAuth entirely, since the cookie hadn't propagated by onMount), then moved it to save-time, then gave up auto-redirecting and showed an amber "try another browser" banner. The actual fix was architectural: switch from `createOAuth2Session` to `createOAuth2Token` (RFC 6749 code exchange), add a server endpoint that validates `userId`+`secret` with the admin SDK and mints a first-party `mm_session` cookie, and move profile saves server-side too. Lesson: a cross-origin session cookie was never going to survive ITP — the client SDK had to stop being the auth boundary.

## 2026-05-09 — Seeded tasks weren't showing on prod because two env vars were undocumented #incident

The deployed feed was empty even though eight demo tasks lived in Appwrite Cloud. Turned out `APPWRITE_DB_ID` and `APPWRITE_TASKS_TABLE_ID` were never in `.env.example`, so `getTasks()` silently fell through to the in-memory task store — which is empty after every cold start on Vercel. Documented them (`micromatch` / `tasks` from `appwrite.config.json`) and wrote `scripts/seed.ts` to be idempotent by construction: users matched by email, prefs read-modify-write so other keys survive, membership swallows 409, tasks skipped if the title already exists for the org. The seed tasks deliberately carry no deadlines and no relative copy ("this Friday") so the landing page never goes stale.

## 2026-05-08 — Vitest bootstrap broke Vercel deploys via a codepath local builds never hit #incident

Adding Vitest perturbed the dependency graph just enough that `adapter-auto`'s lazy detect-and-import for Vercel resolved a stale `estree-walker` with no `"."` export, failing every deploy with `No "exports" main defined`. Local `bun run build` stayed green the whole time because it skips Vercel detection entirely — so the break was invisible until it hit CI. Fix: drop `adapter-auto` for `adapter-vercel` directly (resolution at install time, not build time) and pin `nodejs22.x` in `svelte.config.js`, because Bun's compat layer reports Node v25.x and adapter-vercel rejects that without an explicit runtime. The takeaway: "works locally" means nothing when the failing codepath only fires under the host's environment detection.

## 2026-05-08 — Test suite caught a real millisecond-collision bug in the dev fallback #milestone #incident

Bootstrapped Vitest and wrote coverage for everything shipped over the prior weeks — tagColors, propublica, email, verifications, badgeDefs, badgeCriteria, badgeAwarder. The suite immediately earned its keep: the in-memory IDs in `verifications.ts` and `badgeDefs.ts` were generated as `mem-${Date.now()}` and collided when two rows were created in the same millisecond, silently overwriting each other. Only affects the Appwrite-less dev fallback, but it was a genuine correctness bug a human would never have noticed by clicking around. Replaced with a counter-suffixed generator. Later the component tests forced migrating to Vitest's `projects` config — `@testing-library/svelte` kept resolving Svelte 5's SSR entry instead of the client one, throwing `lifecycle_function_unavailable`, until each project got the module resolution it needed (node for server, jsdom + `browser` conditions for components). Landed at 123 tests, svelte-check at 0 errors.

## 2026-05-08 — Verification is a soft gate, and clients can't claim it anymore #decision

Built the NGO verification flow as a *soft* gate: NGOs submit org name, country, tax ID, and an optional doc; admins (gated by Appwrite Teams membership) review a queue enriched with ProPublica 501(c)(3) lookups so they can eyeball the EIN against the claimed org name. The non-obvious call: tasks now derive `isVerified` server-side from the verification record, so a client can no longer mark its own tasks verified. Approval back-fills the Verified chip onto every existing task the NGO owns; rejection clears it; both email the NGO via Mailgun's HTTP API (no SDK dependency). Same batch moved badge definitions out of a hardcoded array into an org-owned Appwrite table, and shifted badge awarding from claim *creation* to claim *approval* — because that's when the work is actually verified done — deduped by label so the same badge can't be minted twice.

## 2026-05-08 — Ripped out the Azure HelpBot; it was scaffolding that never earned its slot #pivot

Removed the Azure Direct Line HelpBot widget entirely. `/api/bot/token` now returns 410 Gone for any straggler caller, and the auth pages stopped defaulting to the bot's `help.lottie` mascot. ## 2025-08-14 — MVP in a week: browse → pick → learn → complete → earn #milestone

Shipped the first working MVP three days after the initial commit, framed around a five-verb loop: Browse → Pick → Learn → Complete → Earn. The original pitch leaned hard on just-in-time learning — each task was meant to carry DataCamp and Educative "learning cards" so a volunteer new to a skill could still contribute — plus auto-translation of task copy to the viewer's language and a gamified XP-bar-plus-badges dashboard. The planned deployment was Heroku for the SvelteKit SSR app and DigitalOcean for Appwrite. Most of that early scope (Heroku, DigitalOcean, the learning cards) didn't survive the 2026 revival; the closed claim/approve/badge loop and Appwrite backend did.
